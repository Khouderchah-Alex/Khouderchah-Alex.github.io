Entity Component System
Part 1 - Overview


Overview

The Entity-Component-System―or ECS―pattern is a software architecture pattern where objects are defined by composition (has-a relationships) as opposed to inheritance (is-a relationships). Quite simply, objects are represented as entities, which have no functionality themselves. In fact, entities are often unsigned integers that merely serve as unique identifiers (UIDs). In order for an object to have functionality, components must be attached to its entity. Components do not implement functionality, however, but rather store the relevant state. The actual implementation of functionality is provided by systems, which use the state provided by components. Thus for a single piece of functionality, the system contains the implementation and global state, while the components contain the per-object state. Example

Suppose that you're building a game using the ECS pattern. In order to draw objects, we need to know:

  1.  what to draw
  2.  where to draw it

These two pieces of state correspond to our DrawComponent and TransformComponent, respectively.

Why not just a DrawComponent?

While the code would still work if the DrawComponent contained both what to draw and where to draw it, this would not be a good design. Imagine what would happen later, if we needed to add 3D sounds to our game. Should we duplicate position/orientation/scale information in the SoundComponent? Should the SoundSystem use information from both the SoundComponent and the DrawComponent? Thinking carefully about what components to create and what they should contain can save a lot of headache in the long-run.
